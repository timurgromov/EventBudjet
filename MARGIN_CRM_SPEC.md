# MARGIN_CRM_SPEC.md

Спецификация внутреннего модуля `Margin CRM`.

UI-название модуля в admin-панели может оставаться `Калькулятор маржи`.
Системное название модуля: `Margin CRM`.

---

## 1. Назначение модуля

`Margin CRM` — это внутренняя подсистема для управления реальными заказами, маржой, выручкой, расходами и финансовой историей клиентов.

Модуль строится внутри существующего admin-интерфейса, но концептуально не относится к публичному `Wedding Calculator`.

---

## 2. Цель

Создать рабочий внутренний инструмент, который позволяет:
- быстро считать экономику конкретного сценария;
- на основе расчёта создавать реальный заказ;
- вести карточку действующего клиента;
- отслеживать выручку, прибыль и маржу по периоду;
- видеть общую управленческую картину по текущему и прошлому году;
- использовать эти данные для принятия решений по скидкам, апсейлам и управлению загрузкой.

---

## 3. Границы модуля

### Входит в модуль

- быстрый калькулятор маржи;
- создание заказа из текущего расчёта;
- список существующих заказов;
- карточка заказа;
- строки доходов и расходов по заказу;
- периодическая сводка по выручке, расходам, прибыли и марже;
- внутренняя финансовая история клиента.

### Не входит в первую версию

- полноценная CRM для переписок и менеджерской воронки;
- сложная платёжная система с этапами оплат;
- счета, акты, документы;
- BI-аналитика с десятками измерений;
- смешивание с моделью `lead`;
- автоматическое создание заказов из каждого лида по умолчанию.

---

## 4. Главный принцип

`Margin CRM` и `Wedding Calculator` — это разные продуктовые контуры.

Нельзя использовать сущность `lead` как основную сущность финансового модуля.

Главная сущность `Margin CRM`:
- `client_order`

Связь с `lead`:
- допустима;
- опциональна;
- не является обязательной для создания заказа.

---

## 5. Основной пользовательский сценарий

1. Пользователь открывает раздел `Калькулятор маржи` в admin-панели.
2. В верхней части страницы вводит финансовые параметры сценария.
3. Сразу видит итог по сценарию:
   - выручка;
   - расходы;
   - прибыль;
   - маржа.
4. На этой же странице видит сводную панель по текущему периоду.
5. Если договор подписан и сценарий становится реальным заказом, пользователь заполняет короткий блок данных заказа.
6. Пользователь нажимает `Создать заказ`.
7. Заказ появляется в общем списке на этой же странице.
8. Пользователь открывает карточку заказа и дальше ведёт его финансовую историю.

---

## 6. Структура интерфейса

Главный экран модуля:
- одна страница `/admin/margin-calculator`

Порядок блоков на странице:
1. `Быстрый расчёт`
2. `Данные заказа`
3. `Итог по сценарию`
4. `Панель периода`
5. `Список заказов`

Детальный экран:
- `/admin/margin-calculator/orders/:orderId`

---

## 7. Быстрый расчёт

Быстрый расчёт остаётся верхним и основным входом в модуль.

Поля:
- `Ведущий + DJ`
- `Оборудование`
- `Доп. часы`
- `Стоимость часа`
- `Выплата DJ`
- `Реклама`
- `Прочие расходы`

Назначение:
- быстро посчитать экономику сценария;
- использовать этот же расчёт как стартовую точку создания заказа.

Старая логика калькулятора не должна ломаться.

---

## 8. Блок данных заказа

Этот блок находится на главной странице под быстрым расчётом.

Поля:
- `Имя клиента`
- `Название события`
- `Дата мероприятия`
- `Дата договора`
- `Источник заявки`
- `Статус заказа`
- `Комментарий`

Кнопки:
- `Сбросить`
- `Создать заказ`

### Минимально обязательные поля первой версии

- `Имя клиента`
- `Дата мероприятия`
- `Дата договора`

Дополнительные поля могут быть опциональными.

---

## 9. Системные поля

Пользователь не должен вводить руками:
- `ID заказа`
- `Дата создания карточки`
- `Дата обновления карточки`
- `Год`
- `lead_id`
- итоговые расчётные поля

Система автоматически генерирует:
- код заказа, например `ORD-2026-001`;
- служебные timestamps;
- агрегаты по заказу.

---

## 10. Итог по сценарию

Сразу после ввода данных пользователь должен видеть:
- `Выручка`
- `Общие расходы`
- `Чистая прибыль`
- `Маржа`

Также выводятся статусы:
- статус прибыли;
- статус маржи.

Цель:
- быстро понять, насколько сценарий коммерчески силён;
- соотнести конкретный сценарий с общей картиной периода.

---

## 11. Панель периода

Панель находится на той же странице под итогом по сценарию.

Фильтры первой версии:
- `Текущий год`
- `Прошлый год`
- `Произвольный период`

В первой версии основная аналитика считается по `дате договора`.

Если в будущем понадобится, можно добавить переключение на `дату мероприятия`, но не в первой версии.

### Метрики панели

- `Общая выручка`
- `Общие расходы`
- `Общая прибыль`
- `Средняя маржа`
- `Количество заказов`
- `Средняя прибыль на заказ`
- `Количество заказов с низкой маржой`

---

## 12. Список заказов

Список находится на той же странице ниже панели периода.

Назначение:
- быстро видеть существующую базу заказов;
- открывать карточку нужного клиента;
- отслеживать общую ситуацию без переходов между экранами.

Поля списка:
- `ID`
- `Клиент`
- `Дата договора`
- `Дата мероприятия`
- `Источник`
- `Выручка`
- `Прибыль`
- `Маржа`
- `Статус`
- `Обновлён`

Список должен поддерживать:
- сортировку по дате обновления или дате договора;
- переход в карточку заказа.

Поиск и расширенные фильтры можно добавить позже.

---

## 13. Карточка заказа

Карточка заказа открывается отдельно от главной страницы.

Содержимое карточки:
- шапка заказа;
- клиент и параметры события;
- финансовая сводка;
- строки доходов;
- строки расходов;
- комментарий;
- история изменений.

Через карточку пользователь должен иметь возможность:
- редактировать базовые данные заказа;
- добавлять апсейлы;
- добавлять новые расходы;
- корректировать суммы;
- обновлять комментарий;
- видеть итоговую прибыль и маржу после изменений.

---

## 14. Статусы заказа

Первая версия:
- `Подписан`
- `В работе`
- `Проведён`
- `Закрыт`
- `Отменён`

Пока не вводить:
- сложные платёжные статусы;
- статусы дебиторки;
- многоступенчатую постпродажную воронку.

---

## 15. Доменная модель

### Основная сущность

`client_order`

Назначение:
- хранит заказ как бизнес-объект;
- хранит шапку клиента/события;
- является родителем для финансовых строк.

### Финансовые строки

`client_order_item`

Назначение:
- хранит строки доходов и расходов;
- является источником правды для расчёта итогов.

Итоги по заказу:
- выручка;
- расходы;
- прибыль;
- маржа;

не должны жить только как вручную вводимые поля.

---

## 16. Правила расчёта

Выручка:
- `Ведущий + DJ`
- `Оборудование`
- `Доп. часы * Стоимость часа`

Расходы:
- `Выплата DJ`
- `Реклама`
- `Прочие расходы`

Прибыль:
- `Выручка - Расходы`

Маржа:
- `(Прибыль / Выручка) * 100`, если выручка > 0

При создании заказа из калькулятора система должна переносить текущий расчёт в стартовые строки заказа.

---

## 17. Связь с Wedding Calculator

Связь с сущностью `lead` допустима, но не обязательна.

Правила:
- заказ можно создать без лида;
- заказ можно позже связать с лидом;
- `lead` не является контейнером финансовой истории заказа;
- годовая аналитика `Margin CRM` не строится по данным свадебного калькулятора.

---

## 18. Технические ограничения первой версии

Первая версия должна быть:
- минимальной по числу сущностей;
- понятной в UI;
- безопасной для текущей архитектуры;
- без поломки существующего свадебного калькулятора;
- без большой CRM-надстройки.

Внедрение должно быть additive:
- новые таблицы;
- новые admin API;
- отдельные frontend-компоненты;
- без разрушения текущего `lead` flow.

---

## 19. Acceptance criteria первой полноценной версии

Считать модуль готовым к первой практической эксплуатации, если:

1. Пользователь может открыть `Калькулятор маржи`.
2. Может посчитать сценарий как раньше.
3. Может заполнить данные заказа и создать заказ.
4. Заказ появляется в списке на той же странице.
5. Заказ участвует в агрегатах панели периода.
6. Пользователь может открыть карточку заказа.
7. Может добавить или изменить строки доходов и расходов.
8. Итоги по заказу пересчитываются корректно.
9. `Wedding Calculator` и его `lead`-логика не ломаются.
