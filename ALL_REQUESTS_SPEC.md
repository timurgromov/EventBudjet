# ALL_REQUESTS_SPEC.md

Спецификация внутреннего раздела `Все заявки`.

Раздел относится к `Admin Platform` и не является частью `Wedding Calculator` или `Margin CRM`.

---

## 1. Назначение

`Все заявки` — компактный внутренний реестр входящих заявок из партнёров, рекламы, вебинаров и личных контактов.

Главная цель первой версии:
- заменить текущую Excel-таблицу;
- сохранить привычную скорость внесения данных;
- не плодить лишние CRM-сущности;
- фиксировать источники через справочник, чтобы статистика не расползалась из-за разных написаний;
- подготовить минимальную основу для будущих Telegram-напоминаний.

---

## 2. Границы

### Входит в первую версию

- отдельная вкладка слева в admin-панели;
- создание заявки вручную;
- компактная таблица заявок;
- редактирование прямо в таблице;
- справочник источников заявок;
- выбор источника из фиксированного списка;
- быстрая аналитика по источникам;
- поиск по источнику и комментарию;
- фильтр по статусу;
- фильтр `Требуют внимания`;
- цвет строки по статусу.

### Не входит в первую версию

- импорт в `Margin CRM`;
- связь с `lead`;
- отдельная карточка клиента;
- отдельные поля под имя, телефон, площадку и ответственного;
- AI-подсказки;
- Telegram-напоминания.

---

## 3. Отличие от других контуров

`incoming_request` не равен `lead`.

`lead` — пользователь публичного свадебного калькулятора в Telegram Mini App.

`incoming_request` — ручная строка внутреннего реестра, близкая к текущему Excel.

`incoming_request` не равен `client_order`.

`client_order` — реальный заказ в `Margin CRM`, который участвует в финансовой картине.

`incoming_request` — потенциальная заявка до решения: в работе, подписалась или отказалась.

`incoming_request_source` — отдельный справочник источников ручных заявок.

Он похож по UX на источники свадебного калькулятора, но не является таблицей `lead_sources`, потому что источники Mini App и источники ручных бизнес-заявок отвечают на разные вопросы.

---

## 4. Поля заявки

Минимальная модель:

- `source_id` — ссылка на фиксированный источник заявки. Обязательное поле для новых заявок.
- `source` — денормализованное название источника для совместимости и простого чтения.
- `event_date` — дата мероприятия. Может быть пустой.
- `last_contact_date` — дата последнего контакта. Может быть пустой.
- `meeting_held` — была ли встреча по заявке. По умолчанию `false`; в форме быстрого добавления не показывается, отмечается вручную при редактировании строки. При смене статуса существующей заявки на `Договор` автоматически становится `true`.
- `comment` — свободный комментарий: имя, телефон, история, договорённости, любые детали.
- `status` — технический статус строки.

## 4.1. Источники заявок

Источник заявки хранится в справочнике `incoming_request_sources`.

Поля источника:

- `name` — человекочитаемое название, например `Московское небо`;
- `description` — опциональный комментарий;
- `is_archived` — архивный источник нельзя выбрать для новой заявки, но старая статистика сохраняется.

UX справочника должен повторять плоскую механику источников свадебного калькулятора: пользователь создаёт конкретный источник, а не выбирает отдельный тип или категорию.

Примеры источников:

- `Московское небо`;
- `Олеся`;
- `Вебинар 2026`;
- `Реклама VK`.

Если в системе уже есть заявки со свободным текстом источника, миграция должна создать по ним источники автоматически и привязать заявки к ним.

Статусы:

- `in_work` — в работе, обычная строка;
- `signed` — подписался, зелёная строка;
- `rejected` — отказ, красная строка.

## 5. UX

Раздел должен выглядеть как плотная рабочая таблица, а не как набор карточек.

Основной экран:

- сверху короткий заголовок и счётчики;
- компактная аналитика по всем заявкам;
- таблица эффективности источников;
- быстрый блок добавления источника;
- строка быстрого добавления;
- поиск, фильтр статуса и фильтр источника;
- таблица с inline-редактированием.

Колонки:

1. `Источник`
2. `Дата`
3. `Последний контакт`
4. `Встреча`
5. `Комментарий`
6. `Статус`
7. `Обновлено`
8. действия

Комментарий остаётся главным полем для живого контекста. Не нужно дробить его на имя, телефон и отдельную историю, пока это не станет реальной болью.

Inline-редактирование в таблице сохраняется автоматически после короткой паузы. Отдельной кнопки `Сохранить` в строке быть не должно, чтобы пользователь не терял правки из-за забытого клика.

Сортировка таблицы:

1. Сначала заявки `В работе`, внутри группы — ближайшие к дате мероприятия выше.
2. Затем заявки `Договор`, внутри группы — ближайшие к дате мероприятия выше.
3. Затем заявки `Отказ`, внутри группы — ближайшие к дате мероприятия выше.
4. Заявки без даты мероприятия внутри своей группы идут ниже заявок с датой.

## 5.1. Аналитика

На экране `Все заявки` нужна компактная статистика:

- всего заявок;
- в работе;
- встречи;
- отказались;
- договор;
- требуют внимания;
- общая конверсия в `Договор`;
- конверсия встреч в `Договор`.

Отдельно показывается таблица по источникам:

| Источник | Всего | В работе | Встречи | Отказ | Договор | Конверсия | Встречи % |
|---|---:|---:|---:|---:|---:|---:|---:|

Конверсия считается как:

`signed_count / total_count * 100`

Конверсия встреч считается как:

`signed_count / meeting_count * 100`

Эта аналитика нужна для ответа на вопросы:

- какие источники дают больше всего заявок;
- какие источники дают больше подписанных мероприятий;
- какие источники дают слабую конверсию;
- где много зависших заявок в работе.

## 6. Будущие Telegram-напоминания

На следующем этапе можно добавить тупую регулярную логику:

Если:
- `status = in_work`;
- `last_contact_date` старше 3-4 дней;

то Telegram-бот отправляет напоминание владельцу.

Пример сообщения:

> Чувак, заявка зависла: источник "От Леси", мероприятие 24.06. Последний контакт был 4 дня назад. Комментарий: "договорились, но не перезвонили".

В первой версии backend уже считает флаг `needs_follow_up`, но сам Telegram-пуш ещё не отправляется.

---

## 7. Будущая интеграция с Margin CRM

Позже можно добавить действие `Создать заказ`, если заявка подписалась.

Это должно быть отдельным явным действием пользователя, а не автоматическим переносом каждой заявки в финансовый модуль.
