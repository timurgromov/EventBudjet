PROJECT_SPEC.md

1. Название проекта

Wedding Lead Intelligence System

Telegram Mini App + Telegram Bot + Backend + Admin System для свадебного калькулятора, прогрева лидов и последующей конверсии в диалог и продажу.

⸻

2. Цель проекта

Построить систему, которая переводит воронку из модели:

трафик → вебинар → прогрев → заявка

в модель:

трафик → калькулятор → сбор данных → прогрев внутри Telegram → Lead Intelligence → личный контакт / вебинар / продажа

Система должна:
	•	снижать стоимость первого касания;
	•	давать пользователю полезный инструмент уже на входе;
	•	собирать структурированные данные о лиде;
	•	сохранять прогресс и историю действий;
	•	давать администратору сигналы для точного и своевременного касания;
	•	служить фундаментом для будущего масштабирования трафика.

⸻

3. Бизнес-контекст

3.1 Исходная проблема

Текущая входная точка через бесплатный вебинар имеет высокий барьер внимания. Пользователь должен выделить время, дойти до просмотра и остаться в контенте достаточно долго. Это повышает стоимость лида и снижает доходимость.

3.2 Новая логика продукта

Вместо того чтобы продавать длинный формат на первом касании, система предлагает пользователю быстрый, прикладной и полезный инструмент: свадебный калькулятор бюджета.

3.3 Роль калькулятора

Калькулятор — это не просто таблица расходов. Это:
	•	лид-магнит;
	•	квалификатор;
	•	источник поведенческих данных;
	•	основание для последующих прогревов;
	•	инструмент партнёрской воронки для ресторанов, площадок и подрядчиков.

⸻

4. Продуктовое видение

4.1 Что строится

Строится система из следующих блоков:
	•	Telegram Bot;
	•	Telegram Mini App;
	•	Backend API;
	•	PostgreSQL база данных;
	•	event logging layer;
	•	delayed messaging layer;
	•	admin web panel;
	•	Telegram admin notifications.

4.2 Для кого продукт

Основные пользователи калькулятора:
	•	невеста;
	•	жених;
	•	мама;
	•	свадебный специалист.

Основной коммерческий фокус — лиды из свадебной ниши, находящиеся в стадии подготовки, сравнения бюджетов, выбора площадки и планирования расходов.

4.3 Что пользователь должен чувствовать
	•	вход лёгкий;
	•	интерфейс понятный;
	•	данные не теряются;
	•	инструмент помогает снизить тревогу;
	•	после калькулятора пользователь получает полезные материалы, а не агрессивную продажу.

6. Этапы проекта

Проект делится на 4 этапа.

Этап 1. MVP калькулятора

Цель: пользователь уже может реально пользоваться калькулятором в Telegram Mini App.

Состав:
	•	Telegram Bot;
	•	Mini App frontend;
	•	backend API;
	•	PostgreSQL;
	•	сохранение пользователей;
	•	сохранение профиля;
	•	сохранение расходов;
	•	сохранение прогресса;
	•	расчёт итоговой сметы;
	•	базовые события;
	•	Telegram auth validation;
	•	минимальная admin panel;
	•	базовые Telegram-уведомления администратору.

Этап 2. Автопрогрев

Цель: автоматизировать отправку полезных материалов и прогревов после использования калькулятора.

Состав:
	•	scheduled messages;
	•	шаблоны сообщений;
	•	очередь отправки;
	•	повторные попытки при ошибках;
	•	лог отправок;
	•	остановка / продолжение цепочек по условиям.

Этап 3. Lead Intelligence System

Цель: превратить калькулятор в систему разведки лидов и точных касаний.

Состав:
	•	расширенная карточка лида;
	•	история действий;
	•	фильтры и статусы;
	•	повторные входы;
	•	поведенческие триггеры;
	•	приоритеты лидов;
	•	расширенные уведомления;
	•	удобная admin navigation.

Этап 4. Traffic & Scale

Цель: масштабировать трафик уже на рабочую систему.

Состав:
	•	рекламный трафик на калькулятор;
	•	партнёрские сценарии;
	•	привязка источника трафика;
	•	UTM tracking;
	•	оценка качества лидов по источникам;
	•	конверсионная аналитика.

8. Главные архитектурные ошибки, которые нужно исключить с самого начала

Ошибка 1. Делать только frontend без нормального backend-фундамента

Нельзя хранить критичную логику только во frontend. Все ключевые операции должны проходить через backend:
	•	создание и обновление лида;
	•	привязка Telegram identity;
	•	сохранение расходов;
	•	расчёт и запись итогов;
	•	запись событий;
	•	запуск прогрева;
	•	создание уведомлений.

Ошибка 2. Не закладывать event layer на этапе 1

Если не писать события с самого начала, на этапе 2–3 не будет поведенческой аналитики.

Нужно с этапа 1 писать:
	•	bot_started;
	•	miniapp_opened;
	•	profile_started;
	•	profile_updated;
	•	profile_completed;
	•	expense_added;
	•	expense_updated;
	•	expense_removed;
	•	budget_calculated;
	•	webinar_clicked;
	•	returned_to_app.

Ошибка 3. Не сохранять прогресс

Если пользователь закрывает приложение и теряет всё, продукт теряет часть ценности. Для свадебной ниши это критично. Прогресс обязателен.

Ошибка 4. Смешивать доменную модель с UI-структурой

База должна описывать доменные сущности, а не повторять структуру экранов 1 в 1.

Ошибка 5. Не разделять lead, events, messages и notifications

Нельзя складывать всё в одну таблицу или два абстрактных JSON-поля. Нужно разделять:
	•	lead data;
	•	expenses;
	•	user identity;
	•	event log;
	•	message queue;
	•	admin notifications.

Ошибка 6. Привязать архитектуру к конкретному провайдеру

Нельзя использовать архитектурные решения, которые не позволят переехать между VPS/облаками без переписывания. Стек должен быть portable.

Ошибка 7. Откладывать Telegram auth validation

Если не валидацию initData делать сразу, потом придётся переписывать аутентификацию и чистить сомнительные данные.

Ошибка 8. Не закладывать admin view на этапе 1

Даже минимальная web-admin нужна сразу. Иначе потом приходится внедрять её в уже хаотичную модель данных.

Ошибка 9. Не отделять phase scope от full vision

Нельзя пытаться реализовать все 4 этапа сразу. Но и нельзя писать spec только под первый этап. Архитектурная карта должна быть на весь проект, а реализация — по этапам.

Ошибка 10. Не учитывать multi-project VPS usage

Сервер и контейнерная структура должны с самого начала допускать несколько проектов на одном VPS.

[Telegram User]
      ↓
[Telegram Bot]
      ↓
[Telegram Mini App Frontend]
      ↓
[Backend API / FastAPI]
      ↓
[PostgreSQL]
      ├─ users
      ├─ leads
      ├─ expenses
      ├─ lead_events
      ├─ scheduled_messages

      ↓

9.1 Ключевой принцип

Frontend отвечает за интерфейс и ввод данных. Backend отвечает за:
	•	идентификацию;
	•	валидацию;
	•	запись данных;
	•	восстановление состояния;
	•	правила прогрева;
	•	события;
	•	уведомления;
	•	интеграции.

10. Базовые пользовательские сценарии

10.1 User Journey MVP
	1.	Пользователь открывает Telegram бота.
	2.	Нажимает кнопку открытия калькулятора.
	3.	Mini App получает initData.
	4.	Backend валидирует пользователя.
	5.	Создаётся или обновляется запись user + lead.
	6.	Пользователь заполняет профиль свадьбы.
	7.	Пользователь указывает расходы.
	8.	Система считает итог.
	9.	Прогресс сохраняется.
	10.	Администратор получает уведомление о ключевом действии.
	11.	Пользователь может вернуться позже и продолжить.

10.2 Return Scenario
	1.	Пользователь повторно открывает Mini App.
	2.	Система находит его по validated Telegram identity.
	3.	Загружает сохранённый lead draft.
	4.	Отображает текущее состояние.
	5.	Записывает событие повторного входа.

10.3 Admin Scenario
	1.	Администратор получает Telegram уведомление.
	2.	При необходимости открывает web-admin.
	3.	Видит карточку лида.
	4.	Оценивает данные и активность.
	5.	Решает, требуется ли личное касание.

12. Структура данных и ключевые принципы модели

12.1 Один пользователь может иметь одну активную lead-карточку в MVP

Для первой версии достаточно допустить один активный свадебный кейс на пользователя.

12.2 Расходы — отдельные строки, а не один JSON blob

Это нужно для:
	•	аналитики по категориям;
	•	изменения отдельных строк;
	•	фильтрации;
	•	сравнения пользователей.

12.3 События пишутся append-only

События не перетираются. Они добавляются в лог.

12.4 Lead status отделяется от message status

Статус лида не должен смешиваться со статусом отправки сообщения.

14. Telegram Mini App auth

14.1 Требование

Использовать только валидированную аутентификацию через Telegram initData.

14.2 Принцип
	1.	Frontend получает Telegram.WebApp.initData.
	2.	Отправляет init payload на backend.
	3.	Backend валидирует подпись по bot token.
	4.	Backend извлекает user info.
	5.	Создаёт или обновляет user record.
	6.	Возвращает app session.

14.3 Почему это критично

Без этого нельзя гарантировать подлинность пользователя и корректную привязку данных.

17. Admin layer

17.1 Минимальный admin scope этапа 1

Web-admin должен содержать:
	•	список лидов;
	•	имя / username;
	•	роль;
	•	город;
	•	дата или сезон;
	•	количество гостей;
	•	total budget;
	•	lead status;
	•	last seen;
	•	source;
	•	переход в карточку лида.

Карточка лида должна содержать:
	•	базовый профиль;
	•	расходы;
	•	итоговую сумму;
	•	последние события.

17.2 Telegram admin notifications

На этапе 1 бот должен уметь уведомлять администратора о следующих событиях:
	•	новый старт;
	•	заполнение профиля;
	•	завершение расчёта.

Дополнительно можно включить:
	•	добавление расхода;
	•	повторный вход;
	•	переход к вебинару.

17.3 Future admin scope

На следующих этапах добавляются:
	•	фильтры;
	•	приоритеты;
	•	поиск;
	•	статусы коммуникации;
	•	отметки hot / warm / cold;
	•	сегменты по источникам трафика.

18. Auto messaging layer

18.1 Цель

18.2 Принцип

Даже если этап 2 ещё не реализуется полностью, таблицы и точки интеграции должны быть предусмотрены в архитектуре этапа 1.

	•	данные пользователя не должны теряться;
	•	приложение должно поднимать состояние после перезапуска;
	•	миграции должны быть воспроизводимыми.

21.2 Простота сопровождения
	•	кодовая база должна быть разделена по слоям;
	•	backend должен иметь явные модули api, services, models, schemas;
	•	фронтенд должен иметь явную структуру экранов и состояния.

21.3 Масштабируемость
	•	возможность добавить новые категории расходов без переписывания ядра;
	•	возможность расширить event model;
	•	возможность подключить новые каналы прогрева;
	•	возможность хранить несколько воронок в будущем.

21.4 Переносимость
	•	отказ от жёсткой привязки к облаку;
	•	инфраструктура должна переезжать между Cloud.ru / Selectel / другим VPS.

22. Phase 1 Detailed Scope

22.1 Что считается завершением этапа 1

Этап 1 считается завершённым, если:
	•	пользователь открывает Telegram бота;

	•	существует минимальная admin web page;

26. Правила проектирования для Cursor и разработки

26.1 Общие правила
	•	не писать времянки, которые ломают будущее расширение;
	•	не смешивать инфраструктурные и доменные решения;
	•	не хранить критичную бизнес-логику только на клиенте;
	•	все изменения БД только через миграции;
	•	все external secrets хранить в environment variables;
	•	сервисы должны запускаться через Docker Compose.

26.2 Backend rules
	•	использовать Pydantic schemas;
	•	разделять API, services, models, repositories;
	•	не писать SQL inline в роутерах;
	•	auth validation выносить в отдельный модуль;
	•	event creation делать централизованно.

26.3 Frontend rules
	•	не хранить единственный источник правды только в local state;
	•	восстановление данных делать через backend;
	•	отдельно хранить UI state и server state;
	•	предусмотреть шаги онбординга и возврат к черновику.

26.4 Infra rules
	•	docker volumes для Postgres;
	•	healthchecks у сервисов;
	•	nginx как reverse proxy;
	•	.env separation по средам;
	•	отдельные compose profiles по необходимости.

27. Риски проекта

27.1 Продуктовые риски
	•	калькулятор может не объяснять ценность прогрева, если сообщения будут слишком ранними или слишком продажными;
	•	вход может быть хорошим, но без точных триггеров админ не получит преимуществ Lead Intelligence.

27.2 Технические риски
	•	невалидная Telegram auth;
	•	неправильное сохранение прогресса;
	•	слабая структура event model;
	•	хаос в статусах;
	•	плохая изоляция между проектами на одном VPS.

27.3 Операционные риски
	•	отсутствие backup;
	•	отсутствие логики retry для сообщений;
	•	перегрузка админа шумными уведомлениями.

29. Финальное архитектурное решение

Для проекта фиксируются следующие решения:
	•	весь проект описывается единым PROJECT_SPEC.md;
	•	реализация ведётся по этапам;
	•	стек: React + TypeScript + Vite + Tailwind + shadcn/ui + FastAPI + PostgreSQL + SQLAlchemy + Alembic + aiogram + Docker;
	•	deploy: VPS + Docker Compose + Nginx + HTTPS;
	•	progress saving обязателен с этапа 1;
	•	Telegram initData validation обязателен с этапа 1;
	•	admin web panel и Telegram notifications должны быть заложены уже в MVP;
	•	event logging layer должен существовать уже в MVP;
	•	auto messaging layer должен быть предусмотрен в модели данных уже в MVP, даже если полноценно будет развёрнут позже.
