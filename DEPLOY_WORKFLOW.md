# DEPLOY_WORKFLOW.md

Постоянная схема работы для проекта `EventBudjet`.

## Цель

Не смешивать:

- локальную разработку;
- прод на VPS;
- Telegram bot polling;
- временные туннели.

## Источник истины

- Код хранится в git/GitHub.
- Локальная машина нужна для разработки и проверки изменений.
- VPS нужен для рабочего прод-запуска.

## Постоянные правила

1. Код меняется локально.
2. Локально бот не запускается.
3. Локально для проверки поднимаются только:

```bash
docker compose up -d frontend backend postgres nginx
```

4. Telegram bot работает только на VPS.
5. Перед прод-обновлением изменения коммитятся в git и пушатся в GitHub.
6. На VPS всегда деплоится код из репозитория, а не ручные несохраненные правки.

## Постоянные команды

Локально поднять проект без `bot`:

```bash
npm run local:up
```

Локально остановить проект:

```bash
npm run local:down
```

Задеплоить на VPS:

```bash
npm run deploy:vps
```

Что делает `deploy:vps` теперь:

1. `git pull` на VPS
2. `docker compose up -d --build --force-recreate frontend backend bot`
3. ожидание `frontend` и `backend` в `healthy`
4. отдельный `docker compose up -d --force-recreate nginx`
5. короткая проверка `docker compose ps`

Это сделано специально: при одном общем `docker compose up ... frontend backend bot nginx` с `--force-recreate` `nginx` периодически оставался в состоянии `Created` и не поднимал `443`.

Посмотреть основные логи VPS:

```bash
npm run logs:vps
```

## Правильный порядок работы

1. Внести изменения локально.
2. Проверить локально без `bot`.
3. Закоммитить изменения:

```bash
git add .
git commit -m "Your message"
git push
```

4. Выполнить:

```bash
npm run deploy:vps
```

5. Проверить:

```bash
npm run logs:vps
```

Если после деплоя админка или фронтенд недоступны снаружи, первой проверкой смотреть, не завис ли `nginx` в `Created`. Исторически это уже происходило несколько раз до перевода деплоя на двухфазный запуск.

Для UI-задач итоговую визуальную проверку считать валидной только на конечном адресе размещения продукта, где его реально открывает пользователь. Локальный preview, `127.0.0.1`, временные окна Playwright и другие локальные адреса можно использовать только как промежуточную разработческую проверку, но не как источник истины для финального вывода.

## Что нельзя делать

- Не запускать локально `bot`, если продовый бот уже работает на VPS.
- Не использовать старые локальные туннели как постоянную прод-схему.
- Не править прод вручную в обход git, если это не аварийный фикс.
- Не держать разные compose-структуры локально и на VPS.

## Когда допустим аварийный фикс на VPS

Только если прод уже сломан и нужен срочный recovery.

После этого обязательно:

1. перенести изменения обратно в локальный репозиторий;
2. закоммитить их;
3. вернуть обычный workflow через git.

## Если меняется токен или доступ

- Обновить `.env` на VPS.
- Перезапустить нужный контейнер:

```bash
sudo docker compose up -d bot
sudo docker compose up -d backend
```

- Если меняется Telegram token, перезапускать и `bot`, и `backend`.

## Если Mini App показывает чёрный экран

Сначала проверить, не упал ли React на старте из-за runtime-ошибки во фронтенде.

Последний реальный инцидент был 3 апреля 2026:

- коммит `36d9a97` убрал `useMemo` из импорта в `src/pages/Index.tsx`;
- сам вызов `useMemo(...)` в этом файле остался;
- в итоге приложение собиралось, но падало при открытии `/` с ошибкой уровня `useMemo is not defined`;
- визуально это выглядело как чёрный экран в Telegram Mini App.

Почему это может повторяться:

- `npm run build` не ловит все runtime-ошибки React;
- Telegram WebView скрывает детали ошибки лучше, чем обычный браузер;
- если править `src/pages/Index.tsx`, `src/App.tsx` или стартовые хуки, можно сломать первый рендер без ошибок на этапе сборки.

Быстрая проверка:

```bash
npm run build
npx eslint src/pages/Index.tsx src/App.tsx src/main.tsx
git diff -- src/pages/Index.tsx src/App.tsx src/main.tsx
```

На что смотреть в первую очередь:

- удалённый импорт `useMemo`, `useEffect`, `useCallback` или другого React hook;
- вызов хука, который остался в файле после рефакторинга импорта;
- изменения в точке входа: `src/main.tsx`, `src/App.tsx`, `src/pages/Index.tsx`.

Если после изменений в админке внезапно сломался весь Mini App, не считать это автоматически проблемой админки: проверить, не был ли задет общий пользовательский экран `src/pages/Index.tsx`.

Если локальная версия и развернутый продукт выглядят по-разному, не делать вывод по локальной версии. Источник истины для финального визуального решения: только конечный адрес, на котором реально размещён продукт.

## Быстрый смысл

- Локально: разработка.
- GitHub: источник кода.
- VPS: рабочий прод.
