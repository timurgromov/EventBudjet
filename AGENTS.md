Читай в таком порядке:

1. `SYSTEM_MAP.md`
2. Если задача про `Wedding Calculator` и lead-generation flow:
   `PROJECT_SPEC.md`

   `MARGIN_CRM_SPEC.md`

Не начинай работу с `README.md`: он остался от Lovable-шаблона и не описывает реальный production flow проекта.

## Базовые правила работы

- Сначала смотри `git status` и не трогай чужие незавершённые изменения.
- Делай минимальные правки без лишнего рефакторинга.
- Не выдумывай новые фичи и не расширяй scope без явного запроса.
- Если задача завязана на бизнес-логику или порядок этапов, сверяйся с `PROJECT_SPEC.md` и `TASKS_1.md`, а не с догадками.
- Сначала определи, к какому контуру относится задача: `Wedding Calculator`, `Margin CRM` или `Admin Platform`.
- Не смешивай `lead` и `client_order`.
- Не используй `PROJECT_SPEC.md` как source of truth для `Margin CRM`.

## Обязательный delivery flow

Для любого изменения, которое должно оказаться в проде, порядок такой:

1. Локальная правка
2. Локальная проверка
3. Commit
4. Push
5. `npm run deploy:vps`
6. Post-deploy проверка логов и healthchecks

Не останавливайся после локального фикса или после `git push`, если задача предполагает живой результат на VPS.

## Что проверять локально

- После фронтенд-изменений минимум: `npm run build` и `npm test`.
- Если трогал входные точки или стартовый рендер, отдельно проверяй `src/pages/Index.tsx`, `src/App.tsx`, `src/main.tsx`.
- Если менял backend или infra, проверяй соответствующие health endpoints и docker compose сценарий.

## Критичные особенности проекта

- Локально не запускай `bot`, пока продовый bot polling работает на VPS. Для локальной проверки поднимай только `frontend backend postgres nginx`.
- У Telegram должен быть только один polling bot на токен. При `Conflict` сначала ищи второй запущенный инстанс, а не баг в логике приложения.
- Если Telegram API начинает работать нестабильно, рано проверяй сеть и `BOT_TELEGRAM_PROXY_URL`. Не откладывай тему SOCKS5/VPS-proxy до конца расследования.
- Mini App может ломаться не на этапе сборки, а на runtime в Telegram WebView. `npm run build` недостаточно как единственная проверка.
- Telegram WebView агрессивно кэширует фронтенд. При симптомах вида "на VPS задеплоено, а в Telegram старая или битая версия" проверь:
  - стабильные имена ассетов в `vite.config.ts`
  - cache-bust параметр `app_v` в `bot/main.py`
  - открытие `/` после деплоя
- Админка и Mini App живут в одном фронтенде. Изменения в admin-части могут случайно уронить главный экран `/`, и наоборот.

## Документация и handoff

- Если меняешь deploy flow, VPS topology, proxy routing, backup/restore или секреты, обновляй соответствующие runbook-файлы в репозитории в той же задаче.
- Не храни реальные токены, пароли, приватные ключи и сертификаты в git.
- Если пришлось сделать аварийный ручной фикс на VPS, верни его обратно в git сразу после восстановления сервиса.

## GitHub-поведение

- Если работа идёт по GitHub issue/PR, оставь короткий статус-комментарий до начала существенных правок и после deploy/verification.
- Не считай задачу завершённой, пока в GitHub не отражён фактический статус: что изменено, что задеплоено, что проверено.
