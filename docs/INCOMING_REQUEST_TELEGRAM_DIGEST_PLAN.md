# Incoming Request Telegram Digest Plan

## Назначение

Внутренний Telegram-дайджест нужен, чтобы не держать в голове заявки со статусом `В работе`.

Первая версия намеренно простая:

- отдельный закрытый Telegram-канал;
- текущий Telegram-бот только отправляет сообщения в этот канал;
- каждые 3 дня отправляется список всех заявок `in_work`;
- `last_contact_date`, голосовые команды, LLM и создание заявок из Telegram не входят в v1.

## Telegram-настройка

1. Создать закрытый Telegram-канал для заявок.
2. Добавить текущего бота свадебного калькулятора в канал.
3. Выдать боту право публиковать сообщения.
4. Получить `chat_id` канала.
5. Записать `chat_id` в `.env` на VPS.

Нужные env-переменные:

```env
INCOMING_REQUEST_DIGEST_ENABLED=true
INCOMING_REQUEST_DIGEST_CHAT_ID=-1000000000000
INCOMING_REQUEST_DIGEST_INTERVAL_DAYS=3
INCOMING_REQUEST_DIGEST_CHECK_INTERVAL_SECONDS=300
INCOMING_REQUEST_DIGEST_SEND_HOUR=11
INCOMING_REQUEST_DIGEST_TIMEZONE=Europe/Moscow
```

## Поведение

Бот периодически проверяет, пора ли отправлять дайджест.

Дайджест считается от последней успешной отправки:

- если успешных отправок ещё не было, первый дайджест уходит после наступления `INCOMING_REQUEST_DIGEST_SEND_HOUR`;
- следующие дайджесты уходят не чаще чем раз в `INCOMING_REQUEST_DIGEST_INTERVAL_DAYS`;
- неуспешные отправки логируются, но не считаются успешным дайджестом.

В сообщение попадают только заявки:

- `status = in_work`.

Сортировка:

1. заявки с ближайшей датой мероприятия;
2. заявки без даты;
3. внутри одинаковых групп более свежие обновления выше.

Формат строки:

```text
1. 19.07.2026, через 69 дн. — Вебинар — встреча была
Комментарий коротко
id 51
```

Если дайджест не помещается в лимит Telegram, бот делит его на несколько сообщений.

## Логирование

Отправки пишутся в таблицу `incoming_request_digest_logs`.

Лог хранит:

- `chat_id`;
- `status`: `sent` или `failed`;
- `requests_count`;
- `error`;
- `sent_at`;
- `created_at`.

Это защищает от повторной отправки после рестарта бота.

## Проверка и деплой

Локально не запускать polling-бот, пока production bot работает на VPS.

Проверки перед деплоем:

```bash
npm run build
npm test
```

После деплоя:

1. Проверить health backend/frontend/bot.
2. Проверить `bot` logs.
3. Включить env-переменные на VPS.
4. Перезапустить `bot`.
5. Проверить, что дайджест пришёл в отдельный канал.

## Следующий этап

После того как дайджест приживётся, можно отдельно проектировать:

- текстовую команду в Telegram для создания заявки;
- голосовые сообщения;
- распознавание аудио;
- LLM-разбор источника, даты и комментария;
- подтверждение перед созданием заявки.
