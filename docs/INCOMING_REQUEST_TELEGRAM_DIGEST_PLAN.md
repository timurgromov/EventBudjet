# Incoming Request Telegram Digest Plan

## Назначение

Внутренний Telegram-дайджест нужен, чтобы не держать в голове заявки со статусом `В работе`.

Первая версия намеренно простая:

- отдельный закрытый Telegram-канал;
- текущий Telegram-бот только отправляет сообщения в этот канал;
- каждые 3 дня отправляется список всех заявок `in_work`;
- `last_contact_date`, голосовые команды, LLM и создание заявок из Telegram не входят в v1.

## Архитектура Telegram

Важно: в Telegram есть один bot token, но несколько разных адресатов.

Текущий bot token принадлежит боту свадебного калькулятора. Этот бот может писать в разные чаты, если ему разрешили:

```text
TELEGRAM_BOT_TOKEN
  -> личные сообщения клиентам Mini App
  -> BOT_ADMIN_CHAT_ID: текущие admin notifications по свадебному калькулятору
  -> INCOMING_REQUEST_DIGEST_CHAT_ID: отдельный канал CRM-заявок
```

Это не смешивает клиентский бот, admin notifications и CRM-дайджест. Разделение происходит по `chat_id`:

- клиентские сообщения уходят конкретным пользователям;
- текущие уведомления калькулятора уходят в `BOT_ADMIN_CHAT_ID`;
- дайджест заявок уходит только в `INCOMING_REQUEST_DIGEST_CHAT_ID`.

Для v1 отдельный CRM-бот не нужен. Отдельный бот можно рассмотреть позже, если появятся входящие команды, голосовые заявки или отдельная CRM-переписка.

## Telegram-настройка канала

1. Создать в Telegram новый закрытый канал, например `CRM заявки`.
2. Добавить в канал текущего бота свадебного калькулятора: `@gromov_wedding_bot`.
3. Выдать боту право публиковать сообщения.
4. Получить `chat_id` канала.
5. Записать `chat_id` в `.env` на VPS как `INCOMING_REQUEST_DIGEST_CHAT_ID`.
6. Включить `INCOMING_REQUEST_DIGEST_ENABLED=true`.
7. Перезапустить только сервис `bot`.

### Как получить `chat_id`

Самый простой способ:

1. Добавить в канал временного диагностического бота, например `@RawDataBot`.
2. Написать любое сообщение в канал.
3. Найти в ответе поле `chat.id`.
4. Скопировать значение целиком. Для приватного канала оно обычно выглядит как `-1001234567890`.
5. Удалить диагностического бота из канала.

Альтернатива для Telegram Desktop/Web:

1. Опубликовать сообщение в приватном канале.
2. Скопировать ссылку на сообщение.
3. Если ссылка выглядит как `https://t.me/c/1234567890/5`, то `chat_id` будет `-1001234567890`.

В `.env` нужно писать именно полное значение с минусом и `100`.

Нужные env-переменные:

```env
INCOMING_REQUEST_DIGEST_ENABLED=true
INCOMING_REQUEST_DIGEST_CHAT_ID=-1000000000000
INCOMING_REQUEST_DIGEST_INTERVAL_DAYS=3
INCOMING_REQUEST_DIGEST_CHECK_INTERVAL_SECONDS=300
INCOMING_REQUEST_DIGEST_SEND_HOUR=11
INCOMING_REQUEST_DIGEST_TIMEZONE=Europe/Moscow
```

После изменения `.env` на VPS:

```bash
cd /home/user1/EventBudjet
sudo docker compose up -d bot
sudo docker compose logs --tail=100 bot
```

## Поведение

Бот периодически проверяет, пора ли отправлять дайджест.

Дайджест считается от последней успешной отправки:

- если успешных отправок ещё не было, первый дайджест уходит после наступления `INCOMING_REQUEST_DIGEST_SEND_HOUR`;
- следующие дайджесты уходят не чаще чем раз в `INCOMING_REQUEST_DIGEST_INTERVAL_DAYS`;
- неуспешные отправки логируются, но не считаются успешным дайджестом.

В сообщение попадают только заявки:

- `status = in_work`.

Сортировка:

1. заявки с ближайшей датой мероприятия;
2. заявки без даты;
3. внутри одинаковых групп более свежие обновления выше.

Формат строки:

```text
1. 19.07.2026, через 69 дн. — Вебинар — встреча была
Комментарий коротко
id 51
```

Если дайджест не помещается в лимит Telegram, бот делит его на несколько сообщений.

## Логирование

Отправки пишутся в таблицу `incoming_request_digest_logs`.

Лог хранит:

- `chat_id`;
- `status`: `sent` или `failed`;
- `requests_count`;
- `error`;
- `sent_at`;
- `created_at`.

Это защищает от повторной отправки после рестарта бота.

## Проверка и деплой

Локально не запускать polling-бот, пока production bot работает на VPS.

Проверки перед деплоем:

```bash
npm run build
npm test
```

После деплоя:

1. Проверить health backend/frontend/bot.
2. Проверить `bot` logs.
3. Включить env-переменные на VPS.
4. Перезапустить `bot`.
5. Проверить, что дайджест пришёл в отдельный канал.

## Следующий этап

После того как дайджест приживётся, можно отдельно проектировать:

- текстовую команду в Telegram для создания заявки;
- голосовые сообщения;
- распознавание аудио;
- LLM-разбор источника, даты и комментария;
- подтверждение перед созданием заявки.
