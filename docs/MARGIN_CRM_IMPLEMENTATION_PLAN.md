# MARGIN_CRM_IMPLEMENTATION_PLAN.md

Пошаговый план внедрения модуля `Margin CRM`.

Документ описывает порядок реализации так, чтобы:
- не ломать `Wedding Calculator`;
- не смешивать `lead` и `client_order`;
- вводить новый модуль постепенно;
- получать рабочий результат на каждом этапе.

---

## 1. Цель внедрения

Превратить текущий utilitarian `Калькулятор маржи` в внутренний управленческий модуль:
- с быстрым расчётом;
- с созданием заказов;
- со списком существующих заказов;
- с финансовой карточкой клиента;
- с агрегированной панелью по периоду.

---

## 2. Общие принципы внедрения

1. Не менять бизнес-логику `Wedding Calculator`.
2. Не использовать `lead` как основную сущность нового модуля.
3. Делать только additive-изменения в backend и frontend.
4. Сначала ввести новые сущности и API, потом перестраивать UI.
5. Сохранять работоспособность текущего маржинального калькулятора на всём пути внедрения.

---

## 3. Этап 1. Backend foundation

### Цель

Добавить отдельный доменный слой для `Margin CRM`.

### Что сделать

Добавить новые модели:
- `client_orders`
- `client_order_items`

Добавить миграцию:
- создать таблицы;
- добавить индексы;
- добавить внешние ключи;
- не менять существующие таблицы `leads`, `expenses`, `lead_events`.

### Предлагаемая схема

`client_orders`
- `id`
- `lead_id` nullable
- `order_code`
- `client_name`
- `event_title`
- `event_date`
- `contract_date`
- `source`
- `status`
- `comment`
- `created_at`
- `updated_at`

`client_order_items`
- `id`
- `order_id`
- `item_type`
- `category_code`
- `title`
- `amount`
- `position`
- `created_at`
- `updated_at`

### Критерий завершения

- миграции поднимаются с нуля;
- текущая схема проекта не ломается;
- новые таблицы доступны из ORM.

## 4. Этап 2. Backend API

### Цель

Дать admin UI доступ к данным заказов.

### Что сделать

Добавить:
- schemas;
- repository;
- service;
- admin API endpoints.

### Минимальный API первой версии

- `GET /admin/client-orders`
- `POST /admin/client-orders`
- `GET /admin/client-orders/{order_id}`
- `PATCH /admin/client-orders/{order_id}`
- `DELETE /admin/client-orders/{order_id}`

- `POST /admin/client-orders/{order_id}/items`
- `PATCH /admin/client-orders/{order_id}/items/{item_id}`
- `DELETE /admin/client-orders/{order_id}/items/{item_id}`

- `GET /admin/client-orders/summary?date_from=&date_to=`
- `POST /admin/client-orders/from-margin-calculator`

### Особое требование

Endpoint `from-margin-calculator` должен:
- принимать данные формы;
- создавать заказ;
- переносить текущий расчёт в стартовые строки доходов и расходов;
- возвращать готовый заказ для интерфейса.

### Критерий завершения

- admin API отвечает корректно;
- заказ можно создать, прочитать, обновить и удалить;
- summary отдаёт агрегированные показатели по диапазону дат.

## 7. Этап 5. Карточка заказа

### Цель

Дать возможность вести финансовую историю конкретного клиента.

### Что сделать

Добавить отдельный экран:
- `/admin/margin-calculator/orders/:orderId`

Функции карточки:
- просмотр шапки заказа;
- редактирование данных заказа;
- просмотр и редактирование строк доходов;
- просмотр и редактирование строк расходов;
- комментарий;
- пересчёт итогов.

### Рекомендуемые блоки

- `OrderHeader`
- `OrderFinancialSummary`
- `OrderRevenueItems`
- `OrderCostItems`
- `OrderComment`

### Критерий завершения

- после открытия карточки пользователь может реально сопровождать заказ;
- апсейлы и новые расходы пересчитывают итоговую экономику.

### Цель

Сделать модуль управленческим инструментом, а не только формой ввода.

### Что сделать

- фильтр `Текущий год`

- новый заказ сразу отражается в агрегатах.

## 9. Этап 7. Связь с lead

### Цель

Аккуратно связать новый модуль с существующим wedding lead flow, не смешивая их.

### Что сделать

Необязательные улучшения следующей итерации:
- поле `lead_id` в заказе;
- сценарий `Создать заказ из лида`;
- быстрый переход из карточки лида в карточку заказа;
- отображение связанного заказа в lead-card.

### Запрет

Нельзя:
- переносить всю финансовую модель в `lead`;
- считать `lead` заменой заказа;
- делать `client_order` обязательным продолжением каждого лида.

### Критерий завершения

- между сущностями есть связь, но границы модулей остаются чистыми.

## 13. Definition of done для первой usable версии

Первая usable версия `Margin CRM` готова, если:

1. Текущий быстрый расчёт работает.
2. Из него можно создать заказ.
3. Заказ сохраняется в отдельной сущности.
4. Заказ виден в списке на главной странице.
5. Заказ участвует в summary по периоду.
6. Есть карточка заказа с редактированием финансовых строк.
7. Свадебный калькулятор и lead-flow не ломаются.

---

## 14. Что не делать раньше времени

Не добавлять в первую волну:
- сложную CRM-воронку;
- менеджерские статусы коммуникаций;
- полноценную оплатную модель;
- сложные дешборды по месяцам, городам и сегментам;
- многослойную аналитику;
- рефакторинг других модулей "заодно".

Фокус первой версии:
- рабочий расчёт;
- рабочее создание заказа;
- рабочая карточка;
- рабочая годовая/периодическая сводка.
