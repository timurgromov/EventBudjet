# SYSTEM_MAP.md

Карта подсистем проекта `EventBudjet`.

Документ нужен для того, чтобы новые чаты и агенты не смешивали разные продуктовые контуры внутри одного репозитория.

---

## 1. Зачем нужен этот документ

В проекте больше нет одной-единственной сущности "калькулятор".

Сейчас в репозитории живут несколько отдельных подсистем:

1. `Wedding Calculator`
2. `Margin CRM`
3. `Admin Platform`

Они используют общую кодовую базу и общую admin-оболочку, но имеют разные цели, разные сущности и разные правила развития.

Если задача относится к одной подсистеме, нельзя автоматически переносить её требования на другую.

---

## 2. Подсистемы

### 2.1. Wedding Calculator

Публичный продукт для лидогенерации и работы с пользователем в Telegram Mini App.

Цель:
- дать пользователю полезный свадебный калькулятор;
- собрать данные о лиде;
- сохранить прогресс;
- передать данные в admin-контур;
- поддержать дальнейший прогрев и конверсию.

Основные сущности:
- `user`
- `lead`
- `expense`
- `lead_event`
- `scheduled_message`
- `admin_notification`

Source of truth:
- [PROJECT_SPEC.md](/Users/ruslanmamedov/EventBudjet/PROJECT_SPEC.md)
- [TASKS_1.md](/Users/ruslanmamedov/EventBudjet/TASKS_1.md)

Не путать с:
- внутренними финансовыми заказами;
- годовой управленческой аналитикой по реальным клиентам.

---

### 2.2. Margin CRM

Внутренний модуль для управления реальными заказами, маржой, выручкой, расходами и финансовой историей клиентов.

UI-ярлык в admin-интерфейсе может оставаться `Калькулятор маржи`, но на уровне системы это отдельный модуль `Margin CRM`.

Цель:
- быстро считать маржу по сценарию заказа;
- из расчёта создавать реальный заказ;
- вести финансовую историю по клиенту;
- видеть агрегированную картину по периоду;
- использовать модуль как внутренний инструмент управления бизнесом.

Основные сущности:
- `client_order`
- `client_order_item`
- в будущем при необходимости `client_order_event`

Source of truth:
- [MARGIN_CRM_SPEC.md](/Users/ruslanmamedov/EventBudjet/MARGIN_CRM_SPEC.md)
- [docs/MARGIN_CRM_IMPLEMENTATION_PLAN.md](/Users/ruslanmamedov/EventBudjet/docs/MARGIN_CRM_IMPLEMENTATION_PLAN.md)

Не путать с:
- `lead` из свадебного калькулятора;
- маркетинговыми событиями пользователя;
- Telegram Mini App user flow.

---

### 2.3. Admin Platform

Общая внутренняя оболочка, в которой живут оба продуктовых контура.

Цель:
- дать единый admin UI;
- хранить admin token access;
- обеспечивать навигацию между модулями;
- переиспользовать общие UI-компоненты и инфраструктуру.

Что сюда относится:
- layout admin-панели;
- shared navigation;
- общие auth/access-механизмы;
- общие визуальные паттерны.

Что сюда не относится:
- бизнес-логика лидогенерации;
- бизнес-логика финансовых заказов.

## 3. Обязательные правила разделения

### Термины

- `Lead` — только входящий пользователь/лид из свадебного калькулятора.
- `Order` / `Client Order` — только реальный коммерческий заказ в модуле `Margin CRM`.
- `Margin Calculator` — интерфейс модуля `Margin CRM`, а не продолжение модели `lead`.
- `Admin` — оболочка, а не отдельная бизнес-сущность.

### Запрет на смешивание

Нельзя:
- использовать `lead` как замену `client_order`;
- складывать финансовую историю реального клиента в сущность `lead`;
- считать годовую выручку и прибыль по таблицам свадебного калькулятора;
- трактовать `Margin CRM` как "ещё одну вкладку лида".

Можно:
- связывать `client_order` с `lead` опциональным `lead_id`;
- создавать заказ вручную без лида;
- в будущем добавлять сценарий `Создать заказ из лида`.

## 5. Практическое правило для будущих агентов

Перед началом задачи нужно ответить на вопрос:

`Это задача про Wedding Calculator, про Margin CRM или про Admin Platform?`

Пока ответ не дан, нельзя проектировать изменения в моделях и API.

---

## 6. Текущее стратегическое состояние

`Wedding Calculator` уже является отдельным публичным продуктовым контуром.

`Margin CRM` развивается как отдельная внутренняя бизнес-подсистема:
- сначала как утилитарный калькулятор маржи;
- затем как заказный/финансовый модуль;
- дальше как внутренняя lightweight CRM для управления собственными заказами.

Поэтому новые specs для `Margin CRM` должны создаваться отдельно, а не как приложение к основному `PROJECT_SPEC.md`.
